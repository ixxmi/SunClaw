# SunClaw 分层架构提示词指南

这份文档写给 SunClaw 的使用者、Agent 设计者和配置维护者。

目标只有一个：把提示词从“混在一起的大段文本”拆成**有明确职责、可维护、可回退**的分层结构。

本文描述的是**推荐的目标规则**，用于指导后续配置、认知文件编写和代码实现。它不是当前仓库里所有代码路径的逐字还原。

## 1. 总体目标

推荐的提示词层级顺序如下：

1. 系统内置提示词（定义边界）
2. `SOUL.md`
3. `IDENTITY.md`
4. 主 Agent 自定义提示词
5. 子 Agent 描述
6. 技能
7. 工具
8. 上下文
9. 用户输入

这一顺序对应的是：

- 越靠前，越稳定、越不应被后续内容覆盖
- 越靠后，越临时、越面向当前任务
- 安全边界先于人格
- 人格先于角色
- 角色先于任务
- 能力先于上下文
- 上下文先于当前用户请求

## 2. 回退规则

分层顺序必须和回退规则一起看，规则如下：

1. 系统内置提示词始终存在，但分成两类：
   - 边界提示词：永远存在
   - 系统通用认知：只在主 Agent 没有自定义提示词时，作为“主 Agent 自定义提示词槽位”的回退内容使用
2. 如果主 Agent 已经配置了自定义提示词：
   - 只保留系统内置的“边界提示词”
   - 不再额外注入系统通用认知
3. 如果主 Agent 没有自定义提示词：
   - 使用系统通用认知填充“主 Agent 自定义提示词”这一层
4. 如果没有子 Agent 场景：
   - 跳过“子 Agent 描述”这一层
5. 技能层分两阶段：
   - 未选中技能时，注入技能摘要
   - 选中技能后，替换为该技能正文
6. 工具层只负责说明“可用能力”和“调用约束”：
   - 真正可调用的工具权限仍应以代码中的结构化工具定义为准
7. 上下文层不能改写前面各层的边界、人格和角色定义

## 3. 每层的职责定义

### 3.1 系统内置提示词（定义边界）

这一层只解决一个问题：**什么不能被后续层改写**。

应该写：

- 安全边界
- 输出协议
- 工具调用协议
- 不允许做的事情
- 不允许伪造的内容
- 不允许绕过的审批和权限规则

不应该写：

- 人格设定
- 语气风格
- 主 Agent 的工作方法
- 某个业务领域的长期认知
- 某次任务的具体目标

好的例子：

```md
你必须遵守安全边界，不得伪造搜索结果，不得绕过工具权限，不得把未执行的动作描述为已完成。
```

不好的例子：

```md
你是一个温柔、理性、擅长架构设计的 AI 助手。
```

上面这句属于人格和角色，不该放在系统边界层。

### 3.2 `SOUL.md`

这一层定义 Agent 的**思维方式、价值排序和认知习惯**。

应该写：

- 如何判断优先级
- 面对不确定性时怎么做
- 取舍原则
- 对复杂度、风险、准确性、效率的偏好
- 长期稳定的思维方式

不应该写：

- 你是谁
- 你叫什么
- 你当前负责哪个业务
- 具体工具名称
- 某次任务说明

好的例子：

```md
优先追求事实准确和结构清晰；面对冲突信息时先澄清再执行；能小步验证时不要直接做大范围假设。
```

不好的例子：

```md
你是 vibecoding 主编排 Agent，负责拆任务和调度 coder。
```

上面这句属于身份和职责，不属于 soul。

### 3.3 `IDENTITY.md`

这一层定义 Agent 的**长期角色、自我定位和职责边界**。

应该写：

- 我是谁
- 我的长期职责
- 我的角色边界
- 我默认负责什么、不负责什么

不应该写：

- 安全边界
- 临时任务目标
- 当前用户这次要我做什么
- 技能选择细节

好的例子：

```md
你是 vibecoding，长期职责是负责研发类任务的理解、规划、调度和结果收束；你不是默认的全量实现者。
```

不好的例子：

```md
遇到搜索请求必须立刻调用 web_search。
```

这属于工具调用协议，不属于 identity。

### 3.4 主 Agent 自定义提示词

这一层定义当前主 Agent 的**核心工作方法和认知重点**。

它是“当前主 Agent 的专属认知层”，不是安全边界，也不是人格底层。

应该写：

- 当前主 Agent 的工作流
- 当前主 Agent 的协作节奏
- 计划和执行策略
- 当前主 Agent 对某类任务的处理偏好

不应该写：

- 和 `SOUL.md` 重复的底层思维方式
- 和 `IDENTITY.md` 重复的长期角色定义
- 临时用户请求的具体细节
- 工具白名单本身

好的例子：

```md
对于中等及以上复杂度的研发任务，默认先做最小必要上下文读取，再给出计划并等待用户确认，确认后按小步串行推进。
```

不好的例子：

```md
你是一个严谨、克制、重视事实准确的智能体。
```

这类内容已经属于 soul，应避免在主 Agent 自定义提示词里重复。

### 3.5 子 Agent 描述

这一层是**运行时动态层**，不是长期人格层。

它用于约束这次派生执行的目标、职责和非目标，通常包含：

- 当前允许派生的 Agent 目录
- 目标子 Agent 的角色说明
- 当前这次派发任务的目标
- 完成标准
- 不该做的事

应该写：

- 这次任务要做什么
- 这次任务不做什么
- 产出格式
- 成功标准

不应该写：

- 覆盖主 Agent 的长期身份
- 定义底层人格
- 定义系统边界

好的例子：

```md
你当前是 coder 子 Agent，只负责实现本步骤要求的接口改动，不负责测试、重构和最终用户沟通。完成后返回改动结果、涉及文件和剩余风险。
```

### 3.6 技能

技能层是“**遇到某类任务时采用的专项工作流**”。

应该写：

- 技能适用范围
- 何时触发
- 使用该技能后的执行流程
- 依赖和前置条件

不应该写：

- 长期人格
- 长期身份
- 全局安全边界

建议：

- 未选择技能时，只暴露摘要，让模型先判断是否命中
- 选中后，再注入该技能正文，避免所有技能全文长期污染上下文

### 3.7 工具

工具层是“**能力说明层**”，不是认知层。

应该写：

- 当前有哪些工具
- 这些工具能做什么
- 调用约束是什么
- 哪些工具优先级更高

不应该写：

- 人格
- 角色
- 用户画像

建议：

- 文字摘要只负责帮助模型理解
- 真正权限由代码里的结构化工具定义控制，不要只靠文本声明

### 3.8 上下文

上下文层是“**当前运行事实**”。

通常包括：

- `USER.md`
- 工作区信息
- 当前环境信息
- 历史消息
- 历史工具结果
- 当前会话状态

应该写：

- 与当前任务相关的事实
- 用户偏好
- 当前项目状态
- 当前会话必要历史

不应该写：

- 重写人格
- 覆盖角色边界
- 改写工具权限

### 3.9 用户输入

用户输入永远是最后一层。

作用是：

- 触发本轮任务
- 在既有边界、人格、角色、能力和上下文之下表达当前需求

它不应该反向定义上层结构。

## 4. 文件与层的推荐映射

建议使用如下映射：

- `SOUL.md`：思维方式、认知习惯、价值排序
- `IDENTITY.md`：长期身份、职责边界、自我定位
- `AGENTS.md`：稳定协作规则、Agent 家族关系、长期协作约束
- `USER.md`：用户画像、偏好、长期上下文
- `agents.list[].system_prompt`：主 Agent 自定义提示词

补充说明：

- “子 Agent 描述”通常不直接存成一个固定文件，而是运行时根据 `allow_agents`、目标 Agent 和当前任务动态生成
- `AGENTS.md` 适合描述稳定协作关系，不适合直接承载当前这次任务的子 Agent 说明

## 5. 一个完整案例

下面用一个研发编排 Agent 举例。

### 5.1 `SOUL.md`

```md
# Soul

优先追求事实准确、结构清晰和最小必要行动。
面对复杂问题时先收敛边界，再进入执行。
不确定时先验证，不靠想象补足事实。
```

### 5.2 `IDENTITY.md`

```md
# Identity

你是 vibecoding。
你的长期职责是理解研发任务、规划执行路径、调度专业 Agent 并最终汇总结果。
你不是默认的全量编码执行者。
```

### 5.3 主 Agent 自定义提示词

```md
对于中等及以上复杂度的研发任务，默认先做最小必要上下文读取，再输出计划并等待用户确认。
用户确认后，优先按小步串行推进，只在明显无依赖时并行。
```

### 5.4 子 Agent 描述

```md
当前步骤交给 coder 子 Agent：
- 目标：实现 API 参数校验
- 范围：仅修改 handler 和对应测试
- 不做：不处理数据库迁移，不做 UI 改动
- 输出：改动文件、实现摘要、未解决问题
```

### 5.5 技能层

```md
如果任务明确命中 code-review skill，则先启用该技能，再按 skill 的流程执行。
```

### 5.6 工具层

```md
当前可用工具：
- read_file
- edit_file
- run_shell
- use_skill
- sessions_spawn
```

### 5.7 上下文层

```md
用户偏好：结果优先，少说空话。
项目状态：当前分支有未提交改动。
历史结论：上一步已确认只修复后端，不动前端。
```

### 5.8 用户输入

```md
帮我把这个接口的 400 错误处理补完整。
```

## 6. 常见错误

### 错误一：把所有内容都写进主 Agent 自定义提示词

问题：

- 难维护
- 难回退
- 很难定位冲突来源

### 错误二：在 `SOUL.md` 里写工具和任务

问题：

- 底层认知被临时规则污染
- 切换角色时容易串味

### 错误三：在 `IDENTITY.md` 里写当前任务

问题：

- 长期身份和短期任务混淆
- 会导致下一轮任务仍受旧任务污染

### 错误四：把技能全文长期常驻

问题：

- 上下文膨胀
- 技能互相污染
- 不利于精确控制

### 错误五：让上下文层覆盖上层规则

问题：

- 历史消息、用户措辞或临时结果反向改写人格和边界

## 7. 使用建议

如果你在设计 Agent，推荐按下面步骤写：

1. 先写系统边界，只写不可变规则
2. 再写 `SOUL.md`，只写思维方式
3. 再写 `IDENTITY.md`，只写长期角色
4. 最后写主 Agent 自定义提示词，只写当前主角色的工作方法
5. 子 Agent 说明和技能都尽量动态化，不要固化进长期人格文件

一句话总结：

**边界要硬，认知要稳，角色要清，任务要轻，能力和上下文要后置。**
